3. Static Output 플러그인

플러그인으로 WP Static HTML Output을 이용한다. 위와 같이 플러그인 페이지에 가서 “Ad New”버튼을 눌러 WP Static HTML Output 플러그인을 설치한 후, 설치 후에 그 플러그인을 Activate한다.

Activate를 한 후에는 위와 같이 Tools탭에 WP Static HTML Output이라는 메뉴가 생성되어 있다. Generate Zip archive of whole site를 체크한 후에, Base Url에는 루트페이지가 될 주소를 적어준다. (ex. 이 repo페이지 경우에는 https://skku-oss-do.github.io 라고 적는다.)

“Generate”버튼을 누르고 조금 기다린 후에 Download Archive가 생긴다. 그 링크를 누르면 .zip형태로 파일을 다운할 수 있다. 다운한 파일을 압축 해제해서 살펴보면, 상당히 깊은 디렉토리 구조로 되어 있는데, index.html파일이 나올 때까지 들어가면 그 위치가 루트 페이지가 될 위치이다.

AGPL

2016-12-06

GNU Affero GPL v3.0

Affero 일반 공중 라이센스라고 하며 공식적으로는 FSF(자유 소프트웨어 재단)에서 현재 관리하고 있는 Affero GPL와, 비공식적으로 Affero, inc.에서 만든 Affero 라이센스를 의미한다. 2002년 3월에 Affero, inc.에서 GNU 일반 공중 라이센스 버전 2(GPLv2)를 기반으로 하여 Affero 일반 공중 라이센스 버전 1을 만들었다. Affero GPL의 핵심은 응용 프로그램 서비스 제공 업체(ASP)가 기존 GPL의 카피 레프트 조항의 허점을 이용하는 것을 막기 위해 나왔으며 웹 어플리케이션과 같이 AGPL 라이센스를 가지고 있는 것이 네트워크 내에서 서비스 된다면 전체 코드를 제공해야 된다는 것이다.

유래

2000년에 헨리 풀과 리차드 스톨만은 GPL v2의 ASP 허점으로 인해 해결하는 방법을 논의하였다. 2001년 헨리 풀은 Affero, inc.를 설립하였고 Affero의 코드를 사용하여 웹 서비스를 만드는 다른 회사때문에 새로운 라이센스를 필요로 하였다. 2002년 자신의 코드를 출력하는 프로그램인 quine에서 아이디어를 얻었다. 2002년 3월 Affero, Inc.는 Affero 프로젝트에 사용할 AGPL v1를 배포하였다. 자유 소프트웨어 재단은 GPL v3에 AGPL v1의 특별한 규정으 포함하길 원하였지만 각 라이센스 특수성을 인정하여 AGPL, GPL을 독립적으로 배포하였다. 최종적으로 AGPL v1과의 관계로 인해 Affero의 이름을 유지하며 GNU GPL v3와 버전 번호를 맞추어 최종적으로 GUN AGPL v3를 2007년 11월 19에 FSF에 의해 배포하게 된다. AGPL은 GPL v3와 상호 호환성의 형태를 위한 조항(각 라이센스의 13항)을 가지고 있다.

AGPL 특징

AGPL은 GPL 3판에 추가될 내용을 먼저 담고 있는 사용허가서의 하나이며 형식면에서 볼 때 AGPL은 GPL을 그대로 차용한 후에 제2조에 하나의 조항을 추가해서 만든 것이다. 추가된 제2조 4항은 다음과 같다.

제 4 항. 프로그램이 컴퓨터 네트워크를 통해 대화형으로 작동되도록 의도된 것이었고  
양도받은 특정 판(version)이 이를 사용하는 어떠한 사용자에 대해서도 전체 원시 코드의  
전송을 요청할 수 있는 기능을 제공하고 있다면, 프로그램을 개작하거나 2차적 프로그램을  
창작할 때 이러한 기능을 제외시킬 수 없습니다. 또한 개작된 프로그램이나 2차적 프로그램의  
전체 원시 코드의 전송을 HTTP로 즉시 요청할 수 있는 기회를 컴퓨터 네트워크를 통해  
프로그램을 작동시키는 모든 사용자에게 동등하게 제공해야 합니다.

실효성

<예제 상황>

  1. AGPL을 따르는 오픈소스를 링크하여 Linux에서 드라이버를 작성 한다.
  2. 작성된 Linux용 드라이버를 서비스하는 프로그램에서 호출하여 사용 한다.
  3. 네트워크를 통해서 사용자가 서비스하는 프로그램에 접속하여 서비스를 받는다.

위의 경우 소스코드를 라이브러리 형태로 포함만 하였어도 Linux 드라이버 또한 AGPL 저작물이 됩니다. 또한, Linux용 드라이버를 호출하여 사용하는 서비스 프로그램의 경우 AGPL 저작물인 Linux 드라이버를 사용하고 있기 때문에 결합된 프로그램 전체가 AGPL이 적용 됩니다.

결론. 서비스 프로그램 전체가 AGPL을 적용 받습니다.

단점

AGPL은 앞에서 보신 것처럼 제약이 많습니다. AGPL에 조금이라도 관련된 소스를 사용하게 되면 전체 코드를 공개해야 하기 때문입니다.

LGPL

  • LGPL이 나오게 된 계기

상용 Library와 동일한 기능을 제공하는 Library에 GNU와 같은 엄격한 라이센스를 적용하게 되면, 개발자들이 Library의 사용을 안하게 됩니다. 오히려 이미 널리 사용되고 있는 상용 Library와 동일한 기능을 제공하는 Library를 LGPL로 배포하여 그 사용을 장려하고 사실상의 표준으로 유도하는 점과 관련된 다른 오픈 소스 소프트웨어를 보다 더 많이 사용할 수 있도록 하겠다는 것이 LGPL을 만든 FSF의 전략이다.

  • LPGL 규정
  1. 소프트웨어를 배포하는 경우 LGPL에 의해 배포된다는 사실 명시
  2. LGPL Library의 일부를 수정하는 경우 수정한 Library를 LGPL에 의해 소스 코드 공개
  3. LGPL Library에 응용프로그램을 링크시킬(Static과 Dynamic Linking 모두) 경우 해당 응용프로그램의 소스를 공개할 필요 없음. 다만 사용자가 Library 수정 후 동일한 실행 파일을 생성할 수 있도록 Static Linking시에는 응용프로그램의 Object Code를 제공해야 함

1. 개요

  • 공식 홈페이지 : WordPress

2. 역사

워드프레스의 역사는 2001년도 b2/cafelog를 시작으로 2004년도 워드프레스 1.0으로 세상에 알려지기 시작했다. 이후 2005년도 워드프레스 1.5 버전을 제작하였는데, 이 버전부터 테마 구조를 도입, 오픈 소스 테마를 통해 워드프레스가 오픈소스 CMS로써 세상에 확산하게되는 계기가 되었음. 워드프레스는 전체 웹사이트의 1/6, CMS 웹사이트의 약 60%에 사용되고 있다.

3. 구조

기본적으로 PHP와 HTML/CSS를 사용하는 오픈소스 프로젝트이며 구조는 크게 코어, 테마, 그리고 플러그인으로 나눌 수 있다.

3.1 코어

WordPress의 기본 시스템이라고 할 수 있으며 GNU GPL License로 개발되었다. 시스템에 핵심이 되는 함수들과 클래스들이 선언되어 있다.

3.2 테마

테마는 WordPress로 만들어진 웹사이트의 레이아웃이나 디자인, 기능을 동시에 제공한다. 디자인과 레이아웃은 같은 테마에서도 변경이 가능하고 테마의 업그레이드가 존재할 경우에도 디자인의 변경 없이 업그레이드가 가능하다.
테마는 HTML/CSS와 PHP로 작성되어있지만 사용에 있어서 굳이 앞의 언어를 알아야 할 필요는 없다. 하지만 제작을 할 경우는 어느 정도 숙지하고 있어야 한다.

홈페이지에서도 무료 테마가 다운로드 가능하며, 그 외에도 유료, 무료의 테마가 존재한다.

3.3 플러그인

플러그인은 wordpress로 만들어진 웹사이트나 블로그에 기능을 확장하고 추가하도록 해주는 도구이다. wordpress 홈페이지에서 약 48,000 가지의 플러그인이 존재한다. 테마와는 다르게 디자인이나 레이아웃에 변경을 주지 않고 기능만을 추가한다.
홈페이지에서 약 48,000종류의 플러그인을 제공한다^1

플러그인을 활용하여 반응형 웹을 제작하거나 보안성을 확대시키거나 데이터베이스를 제작할 수도 있다.

4. 장점

비전문가나 프로그래밍에 익숙하지 않은 사람도 웹사이트를 구축하는데 크게 어려움 없이 제작이 가능하다. WordPress를 사용하는데 코어 시스템을 알아야 할 필요 없이 다운로드를 받아서 이용만 하면 된다.

다양한 테마와 플러그인들이 유료 외에도 무료로도 충분히 지원된다. 또한 다국어 홈페이지 제작이나 반응형 웹 제작에도 용이하며 WordPress가 오픈소스 CMS다 보니 사용자들이 많아 사용에 문제가 생길 시 도움을 쉽게 받을 수 있다.

비용적인 측면에서는 보통 웹사이트를 외주로 제작하는 경우 백만 단위의 비용이 들어가지만 WordPress를 이용할 경우 약 10만 원 정도의 제작비용만 들인다면 제작이 가능하다.

5. 단점

많은 테마와 플러그인이 있지만 이들 사이에 규격이 정해진 게 아니기 때문에 사용 중 충돌이 일어날 경우가 많다.
또한 테마와 플러그인들이 맞춤형으로 제작되지 않았기 때문에 원하는 서비스를 완전히 충족시키기는 어렵고 그에 대해 커스텀을 하게 될 경우 PHP와 HTML 사용법에 익숙해야 하며 다른 플러그인들과의 연관성도 고려해야 하기 때문에 시간, 비용이 더 투자될 가능성도 있다.

국내에도 WordPress 관련 커뮤니티는 존재하지만 소규모로 대부분의 정보는 영문으로 된 걸 구해야 한다는 것도 하나의 단점이며 블로그 형식의 CMS로 게시판 형식의 웹사이트에 익숙한 한국 사용자에게는 낯설게 보일 수도 있다.

외국에서 만들어진 테마나 플러그인의 경우 한국어 폰트가 제대로 적용되지 않는다는 단점도 존재한다.
